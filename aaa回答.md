# 1.context:一个 session 连续聊了 200 轮，context 快爆了。你会怎么做压缩？如何确保压缩后的对话仍然流畅？

回答：
1.将llm上下文（1M为例）分为4个区域（摘要区59.6%，无损区29.6%，史官区1%，空区10%）；
2.所有对话持久化到磁盘用于回溯；摘要区可能放有压缩精度8k的摘要/压缩精度24k的摘要/.../原始消息；无损区只放有原始消息；史官区放置从开始到现在的决策/规则/知识/关键点/元认知等等；空区保持饥饿；
3.可以想象成一个队列，新加原始消息入口是无损区和史官区的边界；
4.当无损区满了，最先进入无损区的消息进入摘要区；
5.每当处于摘要区的消息达到L1级压缩粒度，生成摘要并附带指向原始消息的ID范围,存在磁盘和内存；同理每当处于摘要区的实际原始消息达到L2级压缩粒度，对原始文档生成摘要并附带指向原始消息的ID范围,存在磁盘和内存......
6.当L1级摘要占满摘要区，替换成L2级更大粒度的摘要；同理......
7.史官区保持全局视野，每次对话动态追加，满了就压缩；
可能的优化:
1.要是第 N 轮和第 N-1 轮的差异很小，新摘要只编码**和之前摘要的不同之处**，而不是重写全文。
2.按知识首次出现的轮次建索引，之后所有对同一文件/函数/变量的引用只保留指针，去重上下文。
3.重要消息（史官区有引用的消息提高优先级、与当前活跃文件相关的消息提高优先级、纯工具输出降低优先级）可以给予用更小粒度压缩的特权。
4.决策函数（根据当前预算，模型能力等）动态调整硬编码的59.6%等。





#  2.memory:和聊天 Agent 熟悉半个月后，用户问了一个以前问过的问题，Agent 如何做 memory 召回更合理？

回答：

1.将历史session消息分段向量化，用户提问时用query embedding召回top-k;
2.每个记忆片段附带时间戳，会话号，主题，次数等；加权重排，半个月前的问题，相似度权重应该高于时间权重；

# 3.task:对于长程任务，大模型执行一段时间可能会忘掉目标，你知道哪些解决方案，有什么优缺？

回答：核心是将“记住目标”的成本从模型转移到工程，现有解决方案：
1.**TODO-list**:每一步维护结构化TODO,完成/新增/更改都要标记：优点是轻量直观好实现，强制agent对齐原始目标；缺点是受上下文限制，agent可能不诚实；
2.**memory file**:将目标，进度写入独立文件，每次读取刷新：优点是不受上下文限制；缺点是读写开销，需要agent自觉查阅
3.**Reflection**:定期暂停，让agent对照原始目标评估当前进度：优点是主动纠偏，发现隐藏误差；缺点是消耗额外token,反思质量不稳定
4.**子agent**:高层目标分解成子任务，每层只关注本层细节：优点是减少单步认知负载，天然结构化；缺点是协调者既要理解全局又要拆解对齐，成为单点故障；
5.**系统提示词**：在每次交互的system prompt中重复核心目标：优点是始终在上下文视野；缺点是占用空间，注意力稀释；无法反映动态变化
6.**状态机**：每一步保存完整状态，失败/偏离时候回滚回复：优点是确定性回滚，缺点是状态爆炸；





# 4.tool/session runtime:如果 session state 为 busy，此时用户又发来新消息，或者异步工具完成事件也到达，runtime 应该如何处理？

回答：我会设置2个队列，steer队列存放用户发的中途干预消息（需要立即中断当前工作）；followup队列放置用户的随时添加的后续任务消息（在当前工作完成后，才顺序执行）；而异步工具完成事件到达，则继续处理followup队列的消息







# 5.agent runtime 架构对比：Claude Code 的工具输出方式和国内 GLM / 豆包等 OpenAI-compatible function calling 有什么不同？他们各自这样设计的优缺点是什么？

回答：Anthropic 的模型在生成时就为 agent 场景优化了：

1.Anthropic的文本和工具调用可以在一个content数组里面；工具结果在content block里面，可以带图片/表格；使用自回归的采样方式，模型写出tool_use token;
优点是多步推理自然一边想一边调用，工具调用结果支持多模态，agent流程连贯。缺点是实现复杂，流式要拼content block;自回归可能产生畸形json;

2.Open-AI-Compatible的文本和工具不可以混在一起，必须在content和tool_calls二选一；工具结果在role:tool的消息块里面，content必须是string;采样方式是约束采样，保证输出合法json;
优点是协议简单易实现，json合法性有保障，生态成熟。缺点是文本和工具调用互斥，多步agent需要自己循环拼接；工具结果只支持纯文本；
